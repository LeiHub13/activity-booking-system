# 项目优化方案

## 1. Redis 可优化方向

### 1.1 当前项目中 Redis 的使用

当前系统已经使用 Redis 做登录态管理：

```text
login:token:{token} -> userId
login:user:{userId} -> 当前有效 token
```

作用：

| 功能 | 说明 |
| --- | --- |
| Token 服务端存储 | JWT 不只依赖自身过期时间，还要在 Redis 中存在才有效 |
| 退出登录失效 | 用户退出登录时删除 Redis 中的 token |
| 单用户当前 token | 同一用户再次登录时覆盖旧 token，旧 token 自动失效 |

后续可以继续围绕“安全、性能、用户体验、高并发”四个方向优化。

## 2. Redis 优化点

### 2.1 登录接口限流

#### 问题

登录接口如果没有限制，可能被恶意频繁请求，存在暴力破解密码的风险。

#### 方案

使用 Redis 统计同一 IP 或同一用户名在短时间内的登录失败次数。

Key 设计：

```text
rate:login:ip:{ip}
rate:login:user:{username}
```

示例规则：

```text
同一 IP 1 分钟最多请求 10 次登录接口
同一用户名 5 分钟内连续失败 5 次，临时限制登录
```

实现思路：

```java
Long count = redisTemplate.opsForValue().increment(key);
if (count == 1) {
    redisTemplate.expire(key, 1, TimeUnit.MINUTES);
}
if (count > 10) {
    throw new BusinessException("请求过于频繁，请稍后再试");
}
```

#### 亮点

```text
通过 Redis 计数器实现接口限流，降低暴力破解和恶意刷接口风险。
```

### 2.2 注册 / 登录验证码

#### 问题

注册和登录接口如果完全开放，容易被脚本批量调用。

#### 方案

验证码生成后存入 Redis，并设置短期过期时间。

Key 设计：

```text
captcha:login:{uuid}
captcha:register:{uuid}
```

过期时间：

```text
3 分钟或 5 分钟
```

流程：

```text
1. 前端请求验证码
2. 后端生成验证码和 uuid
3. 验证码存入 Redis
4. 前端登录或注册时提交 uuid + captcha
5. 后端从 Redis 取出验证码并比对
6. 校验成功后删除验证码，防止重复使用
```

#### 亮点

```text
利用 Redis 的自动过期能力保存短生命周期验证码，提升登录和注册安全性。
```

### 2.3 热门活动排行缓存

#### 问题

热门活动排行需要聚合报名数据，如果管理员频繁刷新，会重复执行统计 SQL。

#### 方案

将热门活动排行结果缓存到 Redis。

Key 设计：

```text
stats:popular-activity-ranking:{limit}
```

过期时间：

```text
30 秒到 1 分钟
```

流程：

```text
1. 查询热门排行接口
2. 先查 Redis 缓存
3. 缓存存在，直接返回
4. 缓存不存在，查询数据库
5. 将结果写入 Redis，并设置短期过期时间
```

适合缓存的接口：

```text
GET /api/admin/activity/popular-ranking
GET /api/admin/activity/registration-stats
```

#### 亮点

```text
对统计类接口增加 Redis 缓存，减少重复聚合查询，提高管理端统计页面响应速度。
```

### 2.4 活动详情缓存

#### 问题

活动详情属于读多写少的数据，用户查看活动列表后可能频繁进入详情页。

#### 方案

对已发布活动详情做缓存。

Key 设计：

```text
activity:detail:{activityId}
```

过期时间：

```text
5 分钟到 10 分钟
```

缓存更新策略：

| 操作 | 缓存处理 |
| --- | --- |
| 查询活动详情 | 先查 Redis，没有再查数据库 |
| 编辑活动 | 删除 `activity:detail:{activityId}` |
| 下线活动 | 删除 `activity:detail:{activityId}` |
| 报名 / 取消 / 审核拒绝 | 删除或更新活动详情缓存中的 `currentCount` |

#### 注意

活动详情包含 `currentCount`，报名人数变化比较频繁。如果不想处理缓存一致性，可以只缓存不包含人数的活动基础信息，或者设置较短过期时间。

### 2.5 未读通知数量缓存

#### 问题

前端页面可能频繁请求未读通知数量。

#### 方案

将当前用户未读通知数量缓存到 Redis。

Key 设计：

```text
notice:unread-count:{userId}
```

缓存失效场景：

| 操作 | 处理 |
| --- | --- |
| 生成新通知 | 删除或增加该用户未读数 |
| 标记单条通知已读 | 删除或减少该用户未读数 |
| 全部标记已读 | 删除或设置为 0 |

#### 亮点

```text
缓存未读通知数量，减少首页和侧边栏等高频组件对数据库的重复查询。
```

### 2.6 防重复提交锁

#### 问题

用户可能连续点击报名按钮、取消按钮，短时间内发出多个重复请求。

#### 方案

使用 Redis 做短时间防重复提交锁。

Key 设计：

```text
lock:registration:{userId}:{activityId}
lock:cancel-registration:{userId}:{registrationId}
```

实现方式：

```java
Boolean locked = redisTemplate.opsForValue().setIfAbsent(key, "1", 5, TimeUnit.SECONDS);
if (!Boolean.TRUE.equals(locked)) {
    throw new BusinessException("操作过于频繁，请稍后再试");
}
```

#### 注意

Redis 锁只能提升体验，不能作为最终一致性保障。最终仍然要依赖：

```text
数据库唯一索引
数据库条件更新
Spring 事务
```

### 2.7 在线用户统计

#### 方案

用户登录时记录在线用户，退出或 token 过期后自动失效。

Key 设计：

```text
online:user:{userId}
```

Value 可以存：

```json
{
  "username": "admin",
  "role": "ROLE_ADMIN",
  "loginTime": "2026-05-27T10:00:00"
}
```

用途：

```text
管理员查看当前在线人数
统计系统活跃情况
```

## 3. Redis 优先级建议

| 优先级 | 优化点 | 推荐原因 |
| --- | --- | --- |
| 高 | 登录接口限流 | 安全亮点明显，实现成本低 |
| 高 | 热门活动排行缓存 | 已有统计接口，适合性能优化展示 |
| 中 | 验证码 | 安全增强，适合登录注册模块 |
| 中 | 未读通知数量缓存 | 用户体验优化 |
| 中 | 防重复提交锁 | 提升体验，但不能替代数据库并发控制 |
| 低 | 活动详情缓存 | 要注意报名人数缓存一致性 |
| 低 | 在线用户统计 | 展示型功能，非核心业务 |

## 4. RabbitMQ 接入目标

### 4.1 为什么引入 RabbitMQ

当前项目中的一些操作并不需要和主业务强绑定，例如：

```text
审核通过后生成站内通知
审核拒绝后生成站内通知
活动下线后通知已报名用户
报名成功后发送确认通知
统计数据异步刷新
```

这些操作适合通过消息队列异步处理。

引入 RabbitMQ 后可以实现：

| 目标 | 说明 |
| --- | --- |
| 异步处理 | 主流程快速返回，通知生成等操作交给消费者处理 |
| 业务解耦 | 报名审核逻辑不直接依赖通知生成逻辑 |
| 削峰填谷 | 高峰期大量报名或审核时，消息进入队列慢慢消费 |
| 失败重试 | 通知生成失败后可以重试或进入死信队列 |
| 扩展能力 | 后续可接短信、邮件、WebSocket 推送等 |

## 5. RabbitMQ 推荐落地场景

### 5.1 优先场景：站内通知异步生成

当前审核通过或拒绝时，业务层直接调用：

```java
noticeService.createRegistrationAuditNotice(...)
```

可以改为：

```text
审核报名状态更新成功
提交事务
发送 RabbitMQ 消息
消费者监听消息
消费者生成站内通知
```

这样审核接口只负责“审核状态变更”，通知生成由消息消费者异步完成。

### 5.2 可扩展场景

| 场景 | 消息事件 |
| --- | --- |
| 报名成功 | `registration.created` |
| 报名取消 | `registration.canceled` |
| 审核通过 | `registration.approved` |
| 审核拒绝 | `registration.rejected` |
| 活动下线 | `activity.closed` |
| 活动报名人数变化 | `activity.count.changed` |

## 6. RabbitMQ 设计方案

### 6.1 交换机设计

使用 Topic Exchange：

```text
activity.booking.exchange
```

原因：

```text
Topic Exchange 支持通过 routing key 区分不同业务事件，后续扩展更方便。
```

### 6.2 队列设计

| 队列 | Routing Key | 作用 |
| --- | --- | --- |
| `notice.registration.queue` | `notice.registration.*` | 处理报名相关通知 |
| `notice.activity.queue` | `notice.activity.*` | 处理活动相关通知 |
| `stats.activity.queue` | `stats.activity.*` | 处理活动统计刷新 |
| `activity.booking.dlq` | 死信队列 | 处理消费失败消息 |

### 6.3 Routing Key 设计

| Routing Key | 说明 |
| --- | --- |
| `notice.registration.approved` | 报名审核通过通知 |
| `notice.registration.rejected` | 报名审核拒绝通知 |
| `notice.registration.created` | 报名提交成功通知 |
| `notice.registration.canceled` | 报名取消通知 |
| `notice.activity.closed` | 活动下线通知 |
| `stats.activity.count-changed` | 活动报名人数变化，触发统计缓存刷新 |

## 7. RabbitMQ 消息格式

建议定义统一事件消息 DTO。

```json
{
  "eventId": "uuid",
  "eventType": "REGISTRATION_APPROVED",
  "userId": 1,
  "activityId": 10,
  "registrationId": 100,
  "title": "报名审核通过",
  "content": "你报名的活动已审核通过，请按时参加。",
  "occurredAt": "2026-05-27T10:00:00"
}
```

字段说明：

| 字段 | 说明 |
| --- | --- |
| eventId | 消息唯一 ID，用于幂等消费 |
| eventType | 事件类型 |
| userId | 接收通知的用户 ID |
| activityId | 活动 ID |
| registrationId | 报名记录 ID |
| title | 通知标题 |
| content | 通知内容 |
| occurredAt | 事件发生时间 |

## 8. RabbitMQ 实现步骤

### 8.1 添加依赖

在 `pom.xml` 中加入：

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-amqp</artifactId>
</dependency>
```

### 8.2 添加配置

在 `application.yaml` 中加入：

```yaml
spring:
  rabbitmq:
    host: localhost
    port: 5672
    username: guest
    password: guest
    publisher-confirm-type: correlated
    publisher-returns: true
    listener:
      simple:
        acknowledge-mode: manual
        retry:
          enabled: true
          max-attempts: 3
          initial-interval: 1000
```

### 8.3 创建 RabbitMQ 配置类

配置内容：

```text
声明 TopicExchange
声明业务队列
声明死信队列
绑定 exchange、queue、routing key
```

建议类名：

```text
RabbitMQConfig
```

建议常量类：

```text
RabbitMQConstants
```

### 8.4 发送消息

审核通过或拒绝后，不直接创建通知，而是发送消息。

推荐方式：

```text
业务事务提交成功后，再发送 MQ 消息
```

原因：

```text
如果数据库事务还没提交就发送消息，后续事务回滚时，消费者可能会生成一条并不存在的业务通知。
```

推荐实现：

```java
TransactionSynchronizationManager.registerSynchronization(new TransactionSynchronization() {
    @Override
    public void afterCommit() {
        rabbitTemplate.convertAndSend(exchange, routingKey, message);
    }
});
```

或者使用 Spring 事件：

```text
业务层发布事件
@TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
监听事务提交后再发送 RabbitMQ 消息
```

### 8.5 消费消息

消费者监听队列：

```java
@RabbitListener(queues = "notice.registration.queue")
public void handleRegistrationNotice(RegistrationNoticeMessage message, Channel channel, Message rawMessage) {
    // 1. 幂等校验
    // 2. 生成站内通知
    // 3. 手动 ack
}
```

消费者逻辑：

```text
1. 根据 eventId 做幂等校验
2. 查询或组装通知内容
3. 插入 notice 表
4. 删除或更新 Redis 未读通知数量缓存
5. 手动 ACK
```

## 9. RabbitMQ 可靠性设计

### 9.1 消息发送可靠性

开启 Publisher Confirm：

```yaml
publisher-confirm-type: correlated
```

作用：

```text
确认消息是否成功到达交换机。
```

开启 Publisher Returns：

```yaml
publisher-returns: true
```

作用：

```text
当消息无法路由到队列时，可以回调处理。
```

### 9.2 消息消费可靠性

使用手动 ACK：

```yaml
acknowledge-mode: manual
```

处理方式：

| 情况 | 处理 |
| --- | --- |
| 消费成功 | 手动 ack |
| 临时异常 | nack 并重试 |
| 多次失败 | 进入死信队列 |

### 9.3 幂等消费

RabbitMQ 可能因为网络或消费失败导致消息重复投递，所以消费者必须支持幂等。

可以使用 Redis 记录已处理事件：

Key 设计：

```text
mq:processed:{eventId}
```

处理逻辑：

```java
Boolean firstProcess = redisTemplate.opsForValue()
        .setIfAbsent("mq:processed:" + eventId, "1", 7, TimeUnit.DAYS);

if (!Boolean.TRUE.equals(firstProcess)) {
    // 消息已经处理过，直接 ack
    return;
}
```

也可以在数据库通知表增加 `event_id` 字段并加唯一索引。

### 9.4 死信队列

如果消息多次消费失败，可以进入死信队列。

死信队列作用：

```text
保留异常消息，方便后续人工排查和补偿处理。
```

## 10. RabbitMQ 和 Redis 的配合

RabbitMQ 和 Redis 可以一起增强系统：

| 场景 | Redis | RabbitMQ |
| --- | --- | --- |
| 审核通知 | 缓存未读通知数量 | 异步生成通知 |
| 热门排行 | 缓存排行结果 | 报名人数变化后异步刷新或删除缓存 |
| 防重复消费 | 记录已处理 eventId | 传递业务事件 |
| 接口限流 | 计数限流 | 不参与 |
| 高峰报名 | 防重复提交锁 | 异步削峰处理非核心任务 |

## 11. 推荐落地顺序

### 第一阶段：低成本优化

```text
1. Redis 登录接口限流
2. Redis 热门活动排行缓存
3. Redis 未读通知数量缓存
```

### 第二阶段：RabbitMQ 基础接入

```text
1. 添加 RabbitMQ 依赖和配置
2. 创建交换机、队列、绑定关系
3. 审核通过 / 拒绝后发送通知消息
4. 消费者异步生成站内通知
```

### 第三阶段：可靠性增强

```text
1. 事务提交后发送消息
2. 消费者手动 ACK
3. Redis eventId 幂等控制
4. 死信队列保存异常消息
```

### 第四阶段：扩展能力

```text
1. 活动下线通知已报名用户
2. 报名成功通知
3. 活动统计缓存异步刷新
4. 后续扩展短信、邮件或 WebSocket 推送
```

## 12. 答辩表述

可以这样说明：

```text
项目中 Redis 主要用于登录 token 管理，后续还可以扩展到接口限流、验证码、热门活动排行缓存、活动详情缓存、未读通知数量缓存和防重复提交等场景。其中接口限流提升系统安全性，统计缓存提升管理端查询性能，防重复提交提升用户操作体验。

RabbitMQ 适合用于系统中的异步通知和事件解耦。例如管理员审核报名后，主业务只负责修改报名状态，通知生成通过 RabbitMQ 异步处理。这样可以降低接口响应时间，并将审核业务和通知业务解耦。为了保证可靠性，可以在事务提交后发送消息，消费者使用手动 ACK、Redis 幂等校验和死信队列，避免消息丢失、重复消费和异常消息无法追踪的问题。
```
