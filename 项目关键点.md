# 项目关键点

## 1. 重复报名问题

### 1.1 问题说明

同一个用户不能对同一个活动重复报名。如果只在业务代码中先查询是否报名，再插入报名记录，在并发请求下可能出现两个请求同时通过查询校验的问题。

例如：

```text
请求 A 查询：用户还没报名
请求 B 查询：用户还没报名
请求 A 插入报名记录
请求 B 插入报名记录
```

这样就可能产生重复报名记录。

### 1.2 解决方案

系统使用“业务校验 + 数据库唯一约束”双重控制。

第一层：业务层先查询用户是否已经报名。

```java
Registration existingRegistration = getUserActivityRegistration(currentUser.getId(), activity.getId());
if (existingRegistration != null) {
    if (REGISTRATION_STATUS_CANCELED.equals(existingRegistration.getStatus())) {
        return reactivateCanceledRegistration(existingRegistration, createRegistrationDTO);
    }
    throw new BusinessException("你已报名该活动");
}
```

第二层：数据库层通过唯一索引兜底。

```sql
UNIQUE KEY uk_user_activity (user_id, activity_id)
```

这样即使并发请求同时通过业务查询，数据库也只允许插入一条 `(user_id, activity_id)` 相同的报名记录。

### 1.3 关键点总结

| 控制方式 | 作用 |
| --- | --- |
| 业务层查询 | 提前发现重复报名，给用户友好提示 |
| 数据库唯一索引 | 并发场景下最终兜底，保证数据不会重复 |
| 捕获 DuplicateKeyException | 数据库拦截重复插入后，转换为业务异常 |

## 2. 超额报名问题

### 2.1 问题说明

活动有最大报名人数 `max_count`，当前占用人数为 `current_count`。如果使用“先查人数，再加人数”的方式，在并发场景下可能出现超额报名。

错误思路：

```java
if (activity.getCurrentCount() < activity.getMaxCount()) {
    activity.setCurrentCount(activity.getCurrentCount() + 1);
}
```

并发下可能出现：

```text
活动只剩 1 个名额
请求 A 查询 current_count < max_count，允许报名
请求 B 查询 current_count < max_count，允许报名
请求 A 更新人数
请求 B 更新人数
最终报名人数超过上限
```

### 2.2 解决方案

系统使用数据库原子条件更新控制名额。

核心 SQL：

```java
@Update("""
        update activity
        set current_count = current_count + 1,
            update_time = now()
        where id = #{activityId}
          and status = 'PUBLISHED'
          and current_count < max_count
        """)
int increaseCurrentCount(@Param("activityId") Long activityId);
```

业务层通过影响行数判断是否抢到名额：

```java
int updated = activityMapper.increaseCurrentCount(activity.getId());
if (updated != 1) {
    throw new BusinessException("活动人数已满或活动不可报名");
}
```

### 2.3 为什么这样可以防止超额

`update activity set current_count = current_count + 1 where current_count < max_count` 是一条数据库更新语句，数据库会保证这条语句的执行具有原子性。

当多个请求同时报名时，数据库会逐个执行更新条件判断：

```text
第 1 个请求：current_count < max_count 成立，更新成功，影响行数为 1
第 2 个请求：如果人数已经满了，current_count < max_count 不成立，影响行数为 0
```

只要影响行数不是 1，就说明没有抢到名额，业务层直接返回失败。

### 2.4 关键点总结

| 控制方式 | 作用 |
| --- | --- |
| 条件更新 `current_count < max_count` | 防止活动人数超过上限 |
| 判断影响行数 | 判断当前请求是否真正抢到名额 |
| 不使用先查后改 | 避免并发下多个请求看到相同旧数据 |

## 3. 报名并发控制

### 3.1 总体思路

报名并发控制不是只控制“报名新增”，还要控制取消报名、审核通过、审核拒绝、取消后重新报名等状态变化。

系统采用：

```text
活动人数变化：由 activity 表条件更新控制
报名状态变化：由 registration 表条件更新控制
跨表一致性：由 @Transactional 事务保证
```

核心原则：

```text
只有状态真的变更成功，才允许修改活动人数。
只要后续步骤失败，事务会回滚前面的更新。
```

## 4. 取消报名并发控制

### 4.1 问题说明

用户可能连续点击两次取消报名。如果每次都直接扣减活动人数，就可能导致 `current_count` 被重复扣减。

### 4.2 解决方案

取消报名时，先使用条件 SQL 修改报名状态：

```java
@Update("""
        update registration
        set status = 'CANCELED',
            update_time = now()
        where id = #{registrationId}
          and user_id = #{userId}
          and status in ('PENDING', 'APPROVED')
        """)
int cancelRegistrationIfAllowed(@Param("registrationId") Long registrationId,
                                @Param("userId") Long userId);
```

业务层逻辑：

```java
int updatedRegistration = registrationMapper.cancelRegistrationIfAllowed(registrationId, currentUser.getId());
if (updatedRegistration != 1) {
    throw new BusinessException("报名状态已变化，不能重复取消");
}

int updatedCount = activityMapper.decreaseCurrentCount(registration.getActivityId());
if (updatedCount != 1) {
    throw new BusinessException("活动报名人数更新失败");
}
```

### 4.3 关键点

只有报名状态从 `PENDING` 或 `APPROVED` 成功变成 `CANCELED` 后，才扣减活动人数。

重复点击取消时：

```text
第 1 次请求：状态符合条件，更新成功，扣减人数
第 2 次请求：状态已经是 CANCELED，不符合条件，更新失败，不扣人数
```

## 5. 审核并发控制

### 5.1 审核通过

审核通过只能从 `PENDING` 状态变为 `APPROVED`。

```java
@Update("""
        update registration
        set status = 'APPROVED',
            audit_user_id = #{auditUserId},
            audit_time = now(),
            update_time = now()
        where id = #{registrationId}
          and status = 'PENDING'
        """)
int approveRegistrationIfPending(@Param("registrationId") Long registrationId,
                                 @Param("auditUserId") Long auditUserId);
```

这样可以防止：

```text
管理员重复点击审核通过
一个管理员点通过，另一个管理员点拒绝
用户取消报名后管理员继续审核通过
```

### 5.2 审核拒绝

审核拒绝只能从 `PENDING` 状态变为 `REJECTED`。

```java
@Update("""
        update registration
        set status = 'REJECTED',
            audit_user_id = #{auditUserId},
            audit_time = now(),
            update_time = now()
        where id = #{registrationId}
          and status = 'PENDING'
        """)
int rejectRegistrationIfPending(@Param("registrationId") Long registrationId,
                                @Param("auditUserId") Long auditUserId);
```

业务层先修改报名状态，成功后才释放活动名额：

```java
int updated = registrationMapper.rejectRegistrationIfPending(registrationId, auditor.getId());
if (updated != 1) {
    throw new BusinessException("报名状态已变化，不能重复审核拒绝");
}

int updatedCount = activityMapper.decreaseCurrentCount(registration.getActivityId());
if (updatedCount != 1) {
    throw new BusinessException("活动报名人数更新失败");
}
```

### 5.3 关键点

审核通过和审核拒绝都必须抢同一个条件：

```sql
where id = ?
  and status = 'PENDING'
```

并发情况下，只有一个请求能成功把 `PENDING` 改成新状态。另一个请求再次执行时，状态已经不是 `PENDING`，影响行数为 0。

## 6. 取消后重新报名并发控制

### 6.1 问题说明

系统允许用户取消报名后重新报名。如果用户重复点击报名按钮，也可能导致活动人数重复增加。

### 6.2 解决方案

重新报名时，先抢活动名额，再用条件 SQL 把报名记录从 `CANCELED` 改回 `PENDING`。

```java
@Update("""
        update registration
        set status = 'PENDING',
            remark = #{remark},
            audit_user_id = null,
            audit_time = null,
            update_time = now()
        where id = #{registrationId}
          and status = 'CANCELED'
        """)
int reactivateCanceledRegistration(@Param("registrationId") Long registrationId,
                                   @Param("remark") String remark);
```

业务层：

```java
int updated = activityMapper.increaseCurrentCount(registration.getActivityId());
if (updated != 1) {
    throw new BusinessException("活动人数已满或活动不可报名");
}

int updatedRegistration = registrationMapper.reactivateCanceledRegistration(
        registration.getId(),
        trimToNull(createRegistrationDTO.getRemark())
);
if (updatedRegistration != 1) {
    throw new BusinessException("报名状态已变化，请刷新后重试");
}
```

如果第二步更新报名状态失败，方法上的事务会回滚第一步加的人数。

## 7. 事务控制

### 7.1 为什么需要事务

报名相关操作通常会同时修改两张表：

```text
activity：维护 current_count
registration：维护报名记录和状态
```

如果只更新其中一张表成功，另一张表失败，就会造成数据不一致。

例如：

```text
活动人数 +1 成功
报名记录插入失败
```

如果没有事务，活动人数就会虚增。

### 7.2 项目中的做法

报名、取消报名、审核通过、审核拒绝都使用：

```java
@Transactional(rollbackFor = Exception.class)
```

作用是：

```text
任意一步抛出异常，前面已经执行的数据库更新都会回滚。
```

### 7.3 最终一致性保证

| 操作 | 状态控制 | 人数控制 | 事务保证 |
| --- | --- | --- | --- |
| 新增报名 | 唯一索引防重复 | `increaseCurrentCount` 防超额 | 插入失败回滚人数 |
| 取消报名 | `cancelRegistrationIfAllowed` 防重复取消 | 取消成功后扣人数 | 扣人数失败回滚状态 |
| 审核通过 | `approveRegistrationIfPending` 防重复审核 | 不改变人数 | 通知失败回滚审核 |
| 审核拒绝 | `rejectRegistrationIfPending` 防重复拒绝 | 拒绝成功后扣人数 | 扣人数失败回滚状态 |
| 取消后重新报名 | `reactivateCanceledRegistration` 防重复恢复 | 先抢名额 | 状态恢复失败回滚人数 |

## 8. 答辩表述

可以这样说明：

```text
本系统在活动报名场景中重点处理了重复报名、超额报名和并发状态变更问题。

重复报名方面，系统先在业务层查询用户是否已有报名记录，同时数据库通过 user_id 和 activity_id 的唯一索引做最终兜底，防止并发插入重复报名。

超额报名方面，系统没有使用先查询人数再更新人数的方式，而是使用数据库条件更新 current_count < max_count 来保证名额扣减的原子性。只有 SQL 影响行数为 1 时，才说明当前请求真正抢到名额。

并发控制方面，系统对取消报名、审核通过、审核拒绝、取消后重新报名都使用了带状态条件的 update 语句。例如审核只能从 PENDING 状态变更，取消只能从 PENDING 或 APPROVED 状态变更。这样即使用户或管理员重复点击，也只有第一次请求能成功修改状态，后续请求不会重复扣减人数或覆盖审核结果。

同时，相关业务方法都使用 Spring 事务控制，确保 activity 表的人数变化和 registration 表的报名状态变化保持一致。
```
